# Agent Execution Patch Promotion Apply Decision

## Status

Decision accepted for a default-off, feature-flagged apply path. This document
defines the requirements for turning a reviewed sandbox staged patch into
workspace files. It does not approve automatic or scheduler-triggered
application.

Current product state:

- the sandbox producer may emit a staged patch source
- Runs detail can summarize source, files, checks, promotion status, linked
  Decision title, artifact summary, and patch preview
- pending `workspace.staged_patch` Decisions can jump back to the Run evidence
- approving a `workspace.staged_patch` Decision is preflight-only by default;
  when `enableSandboxPatchPromotionApply` is enabled, approval can apply the
  reviewed patch through the promotion apply service

## First-Principles Problem

A coding agent becomes useful when it can propose file changes. It becomes
dangerous when those changes leave review and touch the user's workspace.

Therefore patch promotion is not a generic "approve and write" action. It is a
state transition from isolated evidence to host mutation. That transition must
answer four questions every time:

1. Is this still the same patch the user reviewed?
2. Is the workspace still in the state that patch expects?
3. Can Taskplane prove what changed and how to undo or inspect it?
4. Can duplicate approvals, app restarts, or partial failures avoid duplicate
   or half-applied writes?

If any answer is unclear, Taskplane should keep the patch reviewable but refuse
promotion.

## Reference Architecture Lessons

### Pi / OpenClaw

The useful lesson is the compact agent loop, not host authority. The model loop
may produce staged edits, but a Taskplane-owned promotion step must remain a
separate policy-controlled transition.

### OpenHands

Execution belongs behind a sandbox boundary. Promotion is the only bridge from
sandbox output to the selected workspace, so it needs the same explicit backend
and capability thinking as the sandbox itself.

### SWE-agent

Agent-computer interfaces work best when actions are narrow and observable. A
promotion tool should not expose a shell or arbitrary write API; it should
apply one validated staged patch source with a bounded file list.

### Plandex

Pending-change review is the right shape: accumulate changes, let the user
review, then apply an explicit accepted set. Taskplane should keep this model
instead of collapsing staged evidence and workspace mutation into one action.

### LangGraph / Human-in-the-loop Runtimes

Human approval should resume from durable state, not from hidden process memory.
The promotion source, policy snapshot, expected files, and review evidence must
be reloadable after restart.

## Decision

Taskplane should support sandbox patch promotion only through a new
`workspace.promote_staged_patch` internal path with these properties:

- explicit user approval from a `workspace.staged_patch` Decision
- no model-visible direct access to the promotion primitive
- source-bound application from a stored staged patch artifact/source
- pre-apply verification before every write
- audit RunSteps before and after application
- idempotent behavior for repeated approval attempts
- fail-closed behavior when the source, artifact, policy, or workspace state is
  incomplete

The existing `workspace.write_patch` tool remains separate. It applies a
model-proposed patch after a tool-permission checkpoint. The new promotion path
applies a sandbox-produced staged source after a patch-promotion Decision.

## Required Apply Contract

The promotion implementation must have a contract shaped like:

```text
promoteStagedPatch({
  runId,
  taskId,
  checkpointId,
  decisionId,
  artifactId,
  sourceId,
  expectedFiles,
  patchDigest,
  policySnapshot,
})
```

It must return:

```text
{
  status: "applied" | "already_applied" | "blocked",
  touchedFiles,
  auditSummary,
  blockedReasons?
}
```

It must not accept raw model text as authority. Raw text can be display
evidence, but application must come from a validated staged patch source or
artifact payload that was generated by the sandbox patch review chain.

## Pre-Apply Gates

Promotion must block unless all checks pass:

- checkpoint is open and kind is `patch_promotion`
- Decision id matches the checkpoint payload
- payload kind is `patch_promotion`
- descriptor id is `workspace.staged_patch`
- policy snapshot still requires Decision promotion
- artifact/source id exists and belongs to the same run/task
- changed file list is non-empty and workspace-relative
- changed files do not include secrets, `.git`, `node_modules`, or path escapes
- patch content digest matches the digest shown during review
- workspace files still match the expected base content or patch hunks
- no previous successful promotion exists for the same checkpoint/source digest

## Apply Semantics

First implementation should use a conservative all-or-nothing file application:

1. Load validated staged patch content.
2. Re-validate expected files and patch digest.
3. Read current workspace content for all touched files.
4. Compute all resulting file contents in memory.
5. If any hunk or expected base check fails, write nothing.
6. Write files only after every target has been validated.
7. Record a RunStep with touched files, digest, and result.
8. Resolve the checkpoint only after successful write or idempotent
   already-applied detection.

This matches the current `workspace.write_patch` safety posture: no partial
application when a later hunk fails.

## Idempotency

Repeated approval or app restart must not reapply blindly.

The promotion path should create or detect a durable promotion record keyed by:

- checkpoint id
- source id
- artifact id
- patch digest

If the record is already applied and current workspace content matches the
expected promoted result, return `already_applied` and keep the checkpoint
resolved. If the record says applied but workspace content diverged, block and
ask for manual review.

## Rollback Position

The first implementation does not need one-click rollback. It must still keep
enough audit data to support manual inspection:

- touched files
- before/after digest per file
- patch digest
- source id and artifact id
- Decision id and actor action

One-click rollback can be a later feature only after Taskplane has a durable
file snapshot or VCS-aware revert strategy.

## User Experience

The UI should keep the current review-first flow:

```text
Run staged patch review
  -> Decision "查看 Run 证据"
  -> approve Decision
  -> promotion service verifies and applies
  -> Run records promotion result
  -> Task timeline records workspace mutation evidence
```

Before promotion is implemented, the UI wording should continue to say that
approval confirms and closes the checkpoint without auto-writing files.

After promotion is implemented, the Decision page should change the copy only
when the backend capability is available and the checkpoint has complete apply
metadata.

## Implementation Slices

### P1: Promotion Readiness Model

Status: first read-only evaluator implemented.

Add a read-only evaluator that maps a patch-promotion checkpoint to:

- ready to apply
- blocked with reasons
- already resolved
- missing source/artifact metadata

This can power UI copy without writing files.

Current evaluator: `evaluateSandboxPatchPromotionReadiness()`.

Current review-only checkpoint payloads return `missing_apply_metadata` until
future persisted apply metadata such as `expectedFiles` and `patchDigest`
exists.

New sandbox patch review checkpoints now carry `expectedFiles` and a `sha256:`
digest of the sandbox patch artifact diff. Older checkpoints remain valid and
continue to report `missing_apply_metadata`.

### P2: Durable Promotion Record

Status: first repository/table implemented.

Add a repository/table or equivalent persisted event that records source id,
artifact id, digest, touched files, status, and timestamps.

Current persistence: `sandbox_patch_promotions`.

The first record shape stores checkpoint id, run id, task id, artifact id,
source id, Decision id, patch digest, expected files, status, audit summary,
blocked reasons, and applied timestamp. It is intentionally not wired to
Decision approval or file writes yet.

### P3: Apply Service

Status: first domain apply service implemented; Decision/UI integration still
deferred.

Implement `promoteStagedPatch()` behind main-process-only domain code. Reuse
the same structured patch parser and all-or-nothing write posture used by
`workspace.write_patch` where possible.

Current preflight: `SandboxPatchPromotionPreflightService` validates the durable
promotion record, checkpoint payload, and patch artifact agree on ids, expected
files, digest, run, and task ownership. It returns `ready`, `blocked`, or
`already_applied` without reading or writing workspace files. This is the input
gate for the future apply service, not the apply service itself.

Current apply core: `SandboxPatchPromotionApplyService` calls the preflight
gate, parses the sandbox review diff emitted by the staged patch collector,
checks touched files against expected files, verifies current workspace content
still matches the reviewed base, prepares all pending writes before writing any
file, and marks the durable promotion record as `applied` or `blocked`. It is
not yet called by Decision approval or UI flows.

### P4: Decision Approval Integration

Status: first preflight-only integration implemented.

Change `DecisionService` only after P1-P3 exist. Approval should call the
promotion service only when readiness is `ready`; otherwise it should fail
closed with a RunStep explaining why no files were written.

Current behavior: approved `patch_promotion` checkpoints call
`SandboxPatchPromotionPreflightService` before any resume path while
`enableSandboxPatchPromotionApply` is false. With
`TASKPLANE_ENABLE_SANDBOX_PATCH_PROMOTION_APPLY=true`, approval calls
`SandboxPatchPromotionApplyService`; applied and already-applied results resolve
the checkpoint with touched-file evidence, while blocked results create a failed
checkpoint RunStep, mark the Run failed, and record that no workspace files were
written.

### P5: UI Copy Upgrade

Update Decision and Runs pages to distinguish:

- review-only checkpoint
- ready-to-promote checkpoint
- applied promotion
- blocked promotion

## Non-Goals

- no automatic scheduler-triggered patch application
- no model-visible patch promotion tool
- no network-enabled promotion
- no package installation during promotion
- no direct Git commit or push
- no one-click rollback in the first implementation
- no application when source metadata is incomplete

## Open Questions

- Where should the durable promotion record live: a dedicated table, RunStep
  payload, or artifact metadata extension?
- Should patch digest be computed over the full diff preview, normalized file
  contents, or both?
- Should promotion require all requested checks to pass, or can failed checks
  still be user-approved for application?
- Should the first implementation require a clean git worktree, or remain VCS
  agnostic and rely on before/after digests?

## Recommendation

Implement P1 next.

It is the smallest useful step: it improves the UI and backend truth model
without writing workspace files. It also forces Taskplane to define the exact
metadata needed for real promotion before the application path exists.
